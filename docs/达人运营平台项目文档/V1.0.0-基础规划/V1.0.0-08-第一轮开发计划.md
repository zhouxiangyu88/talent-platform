# 达人内容运营平台第一轮开发计划 V1.0

## 1. 第一轮目标

第一轮只做一个最小但完整的闭环：达人管理。

这个闭环完成后，你会看到一件完整的事情是怎么发生的：

1. 用户在页面填写达人信息。
2. 前端把信息发给后端。
3. 后端检查数据是否合规、是否重复。
4. 后端把达人保存进数据库。
5. 页面重新展示最新达人列表。

这就是前端、后端、数据库第一次真正连起来。

## 2. 第一轮做什么

第一轮范围：

| 模块 | 做什么 |
| --- | --- |
| 数据库 | 建立达人表 |
| 后端 | 提供达人新增、达人列表、达人详情、达人编辑 API |
| 前端 | 做达人列表页、新增达人抽屉、编辑达人抽屉、达人详情展示 |
| 规则 | 防止同一个“达人 + 媒体平台”重复录入 |
| 验证 | 保存后刷新页面，达人仍然存在 |

## 3. 第一轮不做什么

为了保持清晰，第一轮暂时不做：

- 项目库。
- 内容库。
- 内容数据回收。
- 首页完整看板。
- 爬虫/API 自动同步。
- 权限管理。
- 登录注册。
- 脚本、demo、审核流程。

这些会在后续轮次逐步加入。

## 4. 第一轮页面范围

### 4.1 达人列表页

页面需要展示：

| 字段 | 说明 |
| --- | --- |
| 达人名称 | 达人昵称或名称 |
| 媒体平台 | 抖音、小红书、B站、视频号等 |
| 达人分类 | 美妆、母婴、游戏、科技等 |
| 粉丝数 | 保存当前最新粉丝数 |
| 负责人 | 公司内部负责人 |
| 联系方式 | 展示主要联系方式 |
| 状态 | 正常、停用 |
| 更新时间 | 最近编辑时间 |

页面需要支持：

- 搜索达人名称。
- 按媒体平台筛选。
- 点击“新增达人”。
- 点击“编辑”。
- 点击“查看详情”。

### 4.2 新增达人抽屉

新增达人需要填写：

| 字段 | 是否必填 |
| --- | --- |
| 达人名称 | 必填 |
| 媒体平台 | 必填 |
| 平台账号 ID | 非必填 |
| 主页链接 | 非必填 |
| 达人分类 | 非必填 |
| 粉丝数 | 非必填 |
| 微信 | 非必填 |
| 手机号 | 非必填 |
| 邮箱 | 非必填 |
| 其他联系方式 | 非必填 |
| 负责人 | 非必填 |
| 状态 | 默认正常 |
| 备注 | 非必填 |

保存时规则：

- 达人名称不能为空。
- 媒体平台不能为空。
- 粉丝数如果填写，必须是数字。
- 同一个“达人名称 + 媒体平台”不能重复。

### 4.3 编辑达人抽屉

编辑达人和新增达人使用同一套字段。

保存时规则：

- 可以修改达人信息。
- 如果修改达人名称或媒体平台，也要重新检查是否重复。
- 编辑后列表和详情页都要更新。

### 4.4 达人详情页

第一轮详情页先展示达人基础信息。

需要展示：

- 达人名称。
- 媒体平台。
- 平台账号 ID。
- 主页链接。
- 达人分类。
- 粉丝数。
- 联系方式。
- 负责人。
- 状态。
- 备注。
- 创建时间。
- 更新时间。

后续第二轮/第三轮做内容库后，这里再增加“该达人发布过的内容”。

## 5. 第一轮后端 API

| 接口 | 方法 | 作用 |
| --- | --- | --- |
| `/api/influencers` | GET | 查询达人列表 |
| `/api/influencers` | POST | 新增达人 |
| `/api/influencers/{id}` | GET | 查询达人详情 |
| `/api/influencers/{id}` | PUT | 编辑达人 |

### 5.1 查询达人列表

支持参数：

| 参数 | 说明 |
| --- | --- |
| `keyword` | 搜索达人名称 |
| `platform` | 按媒体平台筛选 |

返回内容：

- 达人列表。
- 每个达人基础字段。

### 5.2 新增达人

前端提交达人信息。

后端检查：

- 达人名称是否为空。
- 媒体平台是否为空。
- 粉丝数是否为数字。
- 同一平台下是否已有同名达人。

检查通过后：

- 写入数据库。
- 返回新增后的达人信息。

### 5.3 查询达人详情

前端传入达人 ID。

后端返回：

- 该达人完整信息。

如果达人不存在：

- 返回“达人不存在”。

### 5.4 编辑达人

前端传入达人 ID 和修改后的信息。

后端检查：

- 达人是否存在。
- 达人名称是否为空。
- 媒体平台是否为空。
- 如果改了名称或平台，是否和其他达人重复。

检查通过后：

- 更新数据库。
- 返回更新后的达人信息。

## 6. 第一轮数据库

第一轮只需要先确保 `influencers` 表可用。

字段如下：

| 字段名 | 中文含义 |
| --- | --- |
| `id` | 达人 ID |
| `name` | 达人名称 |
| `platform` | 媒体平台 |
| `account_id` | 平台账号 ID |
| `profile_url` | 主页链接 |
| `category` | 达人分类 |
| `followers_count` | 最新粉丝数 |
| `wechat` | 微信 |
| `phone` | 手机号 |
| `email` | 邮箱 |
| `other_contact` | 其他联系方式 |
| `owner` | 负责人 |
| `status` | 状态 |
| `remark` | 备注 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

关键约束：

- `name + platform` 唯一。

## 7. 第一轮验收清单

第一轮完成后，我们按下面清单验收：

| 验收项 | 通过标准 |
| --- | --- |
| 页面能打开 | 本地平台可以正常访问 |
| 达人列表能显示 | 能看到数据库里的达人 |
| 新增达人能保存 | 保存后列表出现新达人 |
| 刷新后数据还在 | 说明数据确实进了数据库 |
| 重复达人会被拦住 | 同名同平台不能重复保存 |
| 编辑达人能生效 | 修改后列表和详情页更新 |
| 搜索能用 | 输入关键词能筛选达人 |
| 平台筛选能用 | 选择平台后只看对应平台 |
| 详情能打开 | 能看到单个达人完整信息 |
| 错误提示清楚 | 用户知道为什么保存失败 |

## 8. 第一轮你会学到什么

这一轮你会理解：

| 知识点 | 用业务语言解释 |
| --- | --- |
| 前端表单 | 用户填写资料的地方 |
| API 请求 | 前端把资料交给后端的方式 |
| 后端校验 | 系统保存前先检查资料是否合规 |
| 数据库表 | 长期保存资料的地方 |
| 唯一约束 | 防止重复录入的规则 |
| 列表查询 | 从数据库拿数据并展示 |
| 编辑更新 | 修改已有资料 |
| 错误提示 | 把后端拒绝保存的原因告诉用户 |

## 9. 第一轮开发顺序

建议实际开发按这个顺序：

1. 检查并整理当前项目结构。
2. 确认数据库文件位置。
3. 建立/调整达人表。
4. 编写达人列表 API。
5. 编写新增达人 API。
6. 编写达人详情 API。
7. 编写编辑达人 API。
8. 做达人列表页面。
9. 做新增达人抽屉。
10. 做编辑达人抽屉。
11. 做达人详情展示。
12. 做搜索和平台筛选。
13. 做重复达人提示。
14. 完整验收第一轮。

## 10. 第一轮完成后的下一步

第一轮完成后，第二轮建议做项目库：

- 新增项目。
- 编辑项目。
- 项目列表。
- 项目详情。

第三轮再做内容库：

- 新增已发布内容。
- 内容关联达人。
- 内容可选关联项目。
- 内容链接防重复。

这样节奏会比较稳，你也能一步步看懂系统是怎么长出来的。
