## 3. 文档文件夹规则

每一次迭代单独一个文件夹，统一放在当前目录下。

文件夹命名格式：

```text
版本号-版本主题
```

示例：

```text
V1.2.1-当前线上稳定版
V1.3.0-数据导出与账号安全
V1.3.1-数据导出修复版
```

每个版本文件夹内可以放：

- 完整 PRD
- 本次需求迭代记录
- 页面说明
- 技术方案
- 测试验收记录
- UI 图片或附件

## 5. 单个文件命名规则

每个版本文件夹内的文档统一按以下格式命名：

```text
版本号-序号-文档主题.md
```

示例：

```text
V1.3.0-00-PRD-数据导出与密码管理.md
V1.3.0-01-需求迭代记录-数据导出.md
V1.3.0-02-测试验收记录.md
```

版本号建议规则：

- `V1.2.1`：当前线上稳定版。
- `V1.3.0`：下一次有明确新功能的版本。
- `V1.3.1`：同一功能版本的小修复。
- 每次上线前，先更新 PRD 和迭代记录，再更新代码版本。

## 6. 迭代维护方式

每次新迭代建议这样维护：

1. 复制上一版完整 PRD 到新版本文件夹。
2. 新 PRD 中保留已有功能，并标记本次新增、调整、暂不做。
3. 新增一份需求迭代记录，只记录本次相对上一版的变化。
4. 需求确认后再开始改代码。
5. 代码完成并测试通过后，提交 GitHub 并打新版本标签。

这样可以区分：

- 完整 PRD：这个版本最终是什么样。
- 需求迭代记录：这次比上个版本多了什么、改了什么。
- Git 标签：代码可以回退到哪个稳定版本。
